\section{Introduction}
\label{sec:ch4_intro}

This chapter presents the system design and architecture of the TIBSA (Threat Intelligence-Based Security Assessment) platform. TIBSA is structured as a full-stack, unified cybersecurity platform that integrates threat modeling, vulnerability scanning, threat intelligence ingestion, malware analysis, and multi-stage incident correlation.

The platform follows a layered, service-oriented architecture designed to separate concerns across frontend presentation, backend business logic, and data persistence tiers. As shown in the design schematics, the system coordinates external APIs (such as VirusTotal for reputations) and local engines (including a Random Forest machine learning classifier, static PE file parsers, and a rules-based threat engine) through an authenticated REST API. Security controls, including JWT verification, rate limiting, and database row-level security, are integrated throughout the architecture.

\subsection{Integrated Services and Core APIs}

To deliver comprehensive threat analysis, TIBSA integrates the following services and core interfaces:

\begin{itemize}
    \item \textbf{VirusTotal API:} Provides IP, domain, and URL reputation scans and file hash lookups.
    \item \textbf{AlienVault OTX \& AbuseIPDB:} Ingests indicators of compromise (IOCs) and IP abuse records to support infrastructure investigations.
    \item \textbf{Shodan Integration:} Queries open ports, service banners, and host metadata for active investigations.
    \item \textbf{OpenRouter \& Gemini APIs:} Powers the interactive security chatbot, generates natural-language threat modeling summaries, and compiles malware analysis narratives.
    \item \textbf{Local Machine Learning Engine:} Runs a local Random Forest classifier (trained on lexical features) to compute URL phishing probabilities.
    \item \textbf{Antivirus / ClamAV Container:} Provides local static file scanning co-located with backend Docker containers.
    \item \textbf{Supabase Platform:} Manages relational PostgreSQL storage, user session state, and JWT-based authentication.
    \item \textbf{Threat Modeling Databases:} Employs local datasets for CAPEC attack patterns, CWE references, and OWASP ASVS compliance controls.
    \item \textbf{Railway \& Vercel:} Railway hosts the containerized FastAPI backend, while Vercel serves the Next.js frontend at the network edge.
\end{itemize}

\label{sec:ch4_overview}

\begin{figure}[H]

The TIBSA platform separates presentation, logic, and data concerns into a seven-layer architecture. As illustrated in the High-Level Architecture Diagram (Figure~\ref{fig:high_level_architecture}), clients interact with the platform via the User Layer, passing requests through Edge security and API Gateways. The Backend Services Layer processes the core business logic, relying on the Intelligence Layer, persistent Data Layer, and Threat Engines to return security verdicts. The deployment is distributed across Vercel (frontend), Railway (backend), and Supabase (database \& authentication) to ensure managed scalability and high availability.

The User Layer, detailed in Figure~\ref{fig:layer1}, represents the human actors and client-side interfaces. \subsection{Layer 2 --- Edge Layer}

\begin{figure}[H]
\centering

\caption{Layer 2 --- Edge Layer Architecture}

The Edge Layer, illustrated in Figure~\ref{fig:layer2}, is managed by Vercel's edge network. Before any HTTP request reaches the application servers, it must pass through edge-layer protections. The diagram shows the sequence of edge components: static asset caching on the CDN, Web Application Firewall (WAF) rule sets blocking common web exploits, volumetric DDoS protection, SSL/TLS certificate termination, load balancing, and a bot detection filter designed to drop automated attacks and scraping scripts.

\subsection{Layer 3 --- Frontend Layer}

\begin{figure}[H]
\centering
\includegraphics[width=1.0\textwidth]{media/3.jpg}
\caption{Layer 3 --- Frontend Layer Architecture}
\label{fig:layer3}

The Frontend Layer (Figure~\ref{fig:layer3}) is built as a Next.js (TypeScript) web application. The diagram details three sub-components:
\begin{itemize}
    \item \textbf{UI Components:} Renders forms, analytics charts, uploader wizard fields, threat maps, audit consoles, and user panels.
    \item \textbf{Frontend Logic:} Implements API helper wrappers, auth interceptors (which attach Supabase JWTs as Bearer tokens to all outgoing requests), role validations, and notifications.
    \item \textbf{Frontend Security:} Manages client-side XSS sanitization, token refresh mechanisms, and data masking prior to UI rendering.
\end{itemize}

\subsection{Layer 4 --- API Gateway and Middleware Layer}

\begin{figure}[H]
\centering
\includegraphics[width=1.0\textwidth]{media/4.jpg}
\caption{Layer 4 --- API Gateway and Middleware Layer Architecture}
\label{fig:layer4}
\end{figure}

The API Gateway layer, shown in Figure~\ref{fig:layer4}, intercepts backend requests via a sequential FastAPI middleware pipeline. Incoming requests pass through:
\begin{enumerate}
    \item \textbf{Auth Filter:} Validates user JWTs against Supabase RS256 JWKS endpoints.
    \item \textbf{Rate Limiter:} Enforces endpoints usage quotas using Redis and SlowAPI.
    \item \textbf{Schema Validator:} Validates request bodies against Pydantic models to reject malformed data.
    \item \textbf{Security Filters:} Blocks common injection vectors (SQLi, XSS, RCE) and SSRF attempts.
\end{enumerate}
Once validated, the request is forwarded to the routing engine.

\subsection{Layer 5 --- Backend Services Layer}

\begin{figure}[H]
\centering
\includegraphics[width=0.85\textwidth]{media/5.jpg}
\caption{Layer 5 --- Backend Services Layer Architecture}
\label{fig:layer5}

The Backend Services Layer, detailed in Figure~\ref{fig:layer5}, implements the core business logic of TIBSA. It consists of modular Python services coordinating domain-specific operations:
\begin{itemize}
    \item \textbf{Authentication \& User Services:} Handles registration, MFA validation, and role assignment.
    \item \textbf{Scan \& Website Scanner Services:} Orchestrates automated scans, fingerprinting, and directory discovery.
    \item \textbf{Threat Modeling Service:} Runs the STRIDE evaluation rules.
    \item \textbf{ML Engine \& Investigation Orchestrator:} Performs ML phishing detection and correlates multi-source intelligence reports.
    \item \textbf{Supporting Services:} Includes the Exporter Service (PDF/JSON reports), Billing Service, and Notification Service (Resend/SendGrid alerts).
\end{itemize}

\subsection{Layer 6a --- Intelligence and Monitoring Layer}

\begin{figure}[H]
\centering
\includegraphics[width=0.85\textwidth]{media/6a.jpg}
\caption{Layer 6a --- Intelligence and Monitoring Layer Architecture}
\label{fig:layer6a}

The Intelligence Layer, shown in Figure~\ref{fig:layer6a}, coordinates asynchronous threat scanning. It executes file signature audits using YARA, extracts binary capabilities with CAPA, queries VirusTotal and AbuseIPDB, checks DNS records, and filters Indicators of Compromise (IOCs). The Attack Path Projection service maps these IOCs to known adversary campaign data to outline potential lateral movements.

\subsection{Layer 6b --- Data Layer}

\begin{figure}[H]
\centering
\includegraphics[width=0.85\textwidth]{media/6b.jpg}
\caption{Layer 6b --- Data Layer Architecture}
\label{fig:layer6b}
\end{figure}

The Data Layer, illustrated in Figure~\ref{fig:layer6b}, provides persistence across four systems:
\begin{itemize}
    \item \textbf{PostgreSQL Database:} Supabase-managed database storing relational entities (users, threat models, scan results, and audits) protected by Row-Level Security (RLS) policies.
    \item \textbf{Document / NoSQL Store:} Manages unstructured IOC data and raw threat feeds.
    \item \textbf{Redis Cache:} Stores rate limit counters and hot session keys.
    \item \textbf{Object Storage:} Stores uploaded binaries, YARA/CAPA output, and PDF reports.
\end{itemize}
All database operations are decoupled using repository classes.

\subsection{Layer 7 --- Threat and Risk Engine}

\begin{figure}[H]
\centering
\includegraphics[width=1.0\textwidth]{media/7.jpg}
\caption{Layer 7 --- Threat and Risk Engine Architecture}
\label{fig:layer7}

The Threat and Risk Engine, detailed in Figure~\ref{fig:layer7}, is the analytical core of TIBSA. It parses technical stack profiles through a seven-step STRIDE pipeline: base threat generation, suppression/injection rules, description deduplication, CAPEC/ASVS metadata enrichment, mitigation assignment, residual risk calculation, and projection onto a 5x5 likelihood-vs-impact risk heatmap.

\subsection{Sequence Diagrams}
\label{subsec:sequence_diagrams}

Sequence diagrams illustrate the chronological message exchanges between application components for primary operational flows.

\subsubsection{Flow 1: Standard Security Investigation}
\label{subsubsec:flow1_standard_investigation}

\begin{sidewaysfigure}
\centering
\includegraphics[width=1.0\textheight,height=0.9\textwidth,keepaspectratio]{media/Flow 1.jpg}
\caption{TIBSA Platform --- Standard Security Investigation Sequence Diagram}
\label{fig:sequence_flow1}
\end{sidewaysfigure}
\clearpage

Figure~\ref{fig:sequence_flow1} models the Standard Security Investigation sequence, showing interactions between the Frontend Application, FastAPI Investigations Router, Investigation Orchestrator, PostgreSQL database, and Exporter Service.

The flow begins when the user submits a URL target. The frontend issues an HTTP POST request to the investigations router endpoint `/api/v1/investigations/start`. The router performs initial URL validation and parameters checking. It then instructs the orchestrator service to insert a pending record into the database, generating a unique investigation ID. The router immediately schedules a background pipeline task using FastAPI's native `BackgroundTasks` handler and returns an HTTP 200 response to the user interface to ensure responsiveness.

In the background, the task executes port checks, HTTP headers audits, configuration analysis, and vulnerability correlations. Findings are saved directly to the database. The frontend application polls `/api/v1/investigations/{id}/status` periodically, rendering the current progress. Once the scan status transitions to completed, the user can trigger a report generation request, which prompts the router to query database records and delegate compile jobs to the Exporter Service, generating a PDF or JSON report for download.

\subsubsection{Flow 2: Infrastructure Threat Investigation (Eight-Stage Pipeline)}
\label{subsubsec:flow2_infrastructure_investigation}

\begin{sidewaysfigure}
\centering
\includegraphics[width=1.0\textheight,height=0.9\textwidth,keepaspectratio]{media/Flow 2.jpg}
\caption{TIBSA Platform --- Infrastructure Threat Investigation Sequence Diagram}
\label{fig:sequence_flow2}
\end{sidewaysfigure}
\clearpage

Figure~\ref{fig:sequence_flow2} details the eight-stage infrastructure investigation pipeline sequence. This asynchronous workflow is coordinated by the `InfraOrchestrator` across multiple sub-services:
\begin{enumerate}
    \item \textbf{Intake:} Receives and classifies network inputs (IP, Domain, or URL) via the Intake Service, returning an HTTP 202 Accepted status.
    \item \textbf{Reputation:} Triggers parallel API reputation checks against AbuseIPDB, URLhaus, ThreatFox, and OTX to determine risk levels.
    \item \textbf{Enrichment:} Gathers physical GeoIP coordinates, WHOIS registration ownership, and SSL certificate chains.
    \item \textbf{Passive DNS:} Queries passive lookup histories to map IP changes over time.
    \item \textbf{Indicators:} Applies rulesets to extract operational security indicators of compromise.
    \item \textbf{Correlation:} Groups anomalies to establish relations with known advanced threats.
    \item \textbf{Risk Scoring:} Computes a composite risk score and maps corresponding risk labels.
    \item \textbf{AI Summary:} Generates a natural-language narrative report via OpenRouter LLM gateways.
\end{enumerate}
Upon completing these stages, the orchestrator triggers the Graph Builder service to compile the findings into visual nodes and edges, saving the projected structures in dedicated PostgreSQL tables (\texttt{infra\_indicators}, \texttt{infra\_graph\_nodes}, \texttt{infra\_graph\_edges}, and \texttt{infra\_ai\_reports}) to populate interactive network graphs.

\subsubsection{Flow 3: AI Malware Static Analysis}
\label{subsubsec:flow3_malware_analysis}

\begin{sidewaysfigure}
\centering
\includegraphics[width=0.92\textheight,height=0.82\textwidth,keepaspectratio]{media/Flow 3.jpg}
\caption{TIBSA Platform --- AI Malware Static Analysis Sequence Diagram}
\label{fig:sequence_flow3}
\end{sidewaysfigure}
\clearpage

Figure~\ref{fig:sequence_flow3} details the malware analysis workflow. The user uploads a portable executable (PE) binary file. The API router performs validation checks (ensuring size is under 512MB) and passes the byte stream to the parsing engine. The engine verifies the binary's structural headers and imports.

For valid PE files, the Malware Static Analysis Service extracts binary characteristics: the Import Address Table (IAT) containing external DLL functions, export symbols, section metadata (text, data, rsrc), packing signatures, and entropy levels (detecting obfuscation). These features are passed to a local machine learning classification model to output threat probabilities and classification confidences. The findings are persisted, and the user can request an analyst summary report (compiled via Gemini LLM integration) or download the final formatted PDF document.

Figure~\ref{fig:sequence_flow4} models the execution of independent diagnostic modules. It shows how users can bypass full pipelines to execute standalone tasks:

\end{itemize}

\subsection{Entity Relationship Diagram (ERD)}
\label{subsec:entity_relationship_diagram_erd}

\begin{figure}[H]
\centering
\rotatebox{90}{\includegraphics[width=0.85\textheight,height=\textwidth,keepaspectratio]{media/TIBSA_ERD_full.jpg}}
\caption{TIBSA Platform --- Full Entity Relationship Diagram}
\label{fig:erd_full}
\end{figure}
\clearpage

The Entity Relationship Diagram (Figure~\ref{fig:erd_full}) models the relational database schema implemented in Supabase PostgreSQL:
\begin{itemize}
    \item \textbf{\texttt{users} table:} Anchors user records, authentication credentials, and user roles (admin/user).
    \item \textbf{Investigation tables:} The parent \texttt{investigations} table maintains one-to-many links with the \texttt{findings} table (vulnerability entries) and the \texttt{assets} table (discovered domains/hosts).
    \item \textbf{Infrastructure tables:} The \texttt{infra\_investigations} table maps to projection tables (\texttt{infra\_indicators}, \texttt{infra\_graph\_nodes}, \texttt{infra\_graph\_edges}, and \texttt{infra\_ai\_reports}) to support attack path visualization.
    \item \textbf{Standalone scan tables:} Stores results for file hash checks (\texttt{scans}, \texttt{scan\_reports}) and website vulnerability scans (\texttt{website\_scans}).
    \item \textbf{Threat Modeling table:} The \texttt{threat\_model\_analyses} table stores target profile arrays, STRIDE threat outputs, heatmaps configurations, and ASVS controls.
    \item \textbf{Auditing and Notifications tables:} Logs activity histories (\texttt{audit\_logs}, \texttt{login\_attempts}) and distributes alert messages (\texttt{notifications}).
\end{itemize}

The schema implements foreign key referential integrity with cascade constraints, ensuring parent deletions clean up child findings, while preserving logs and configurations. Database Row-Level Security (RLS) policies restrict table queries to the owner's credentials.

\subsection{Class Diagram}
\label{subsec:class_diagram}

\begin{figure}[H]
\centering
\rotatebox{90}{\includegraphics[width=0.85\textheight,height=\textwidth,keepaspectratio]{media/class_diagram_detailed.jpg}}
\caption{TIBSA Platform --- Detailed Class Diagram}
\label{fig:class_diagram_detailed}
\end{figure}
\clearpage

The Detailed Class Diagram (Figure~\ref{fig:class_diagram_detailed}) models the static backend code layout, organized into the Route Controller layer and the Core Business Services layer.

The \textbf{Route Controller Layer} comprises FastAPI router classes:
\begin{itemize}
    \item \texttt{InvestigationsRouter} handles standard vulnerability scanning triggers and status polling.
    \item \texttt{InfraInvestigationsRouter} exposes endpoints for infrastructure and graph projections.
    \item \texttt{AIMalwareAnalysisRouter} handles multipart file uploads and PE parsing triggers.
    \item \texttt{ThreatModelingRouter} manages STRIDE evaluation inputs.
    \item \texttt{AdminRouter} manages system logs, rates controls, and user states.
\end{itemize}
All routers use `APIRouter` contexts to validate incoming payload schemas against Pydantic models.

The \textbf{Core Business Services Layer} contains service classes encapsulating business logic:
\begin{itemize}
    \item \texttt{InvestigationOrchestrator} runs background scan operations.
    \item \texttt{InfraOrchestrator} implements the Facade pattern, coordinating the intake, reputation, and enrichment sub-services.
    \item \texttt{ProjectionService} distributes the orchestrator's JSONB payloads across relational tables.
    \item \texttt{MalwareStaticService} parses binary headers and calls local machine learning classifier engines.
    \item \texttt{WebsiteScannerService} controls Playwright instances and handles 404 baseline calibrations.
\end{itemize}
Database access is handled through repository classes, decoupling services from the persistent connector layer (`SupabaseClient`).

\subsection{Discussion and Model Validation}
\label{subsec:discussion_and_model_validation}

The system diagrams demonstrate architectural alignment and consistency across all design views:
\begin{itemize}
    \item \textbf{Requirements Traceability:} Use cases (Figure~\ref{fig:use_case_diagram}) trace directly to execution sequences (e.g., UC-02 traces to Flow 1 in Figure~\ref{fig:sequence_flow1}; UC-04 traces to Flow 2 in Figure~\ref{fig:sequence_flow2}).
    \item \textbf{Data Consistency:} All persistent write events shown in the sequence diagrams map directly to SQL entities and columns defined in the ERD (Figure~\ref{fig:erd_full}).
    \item \textbf{Structural Realization:} Sequence diagram components correspond to backend controllers and core service classes mapped in the Class Diagram (Figure~\ref{fig:class_diagram_detailed}), ensuring all execution paths have structural homes.
\end{itemize}

\section{Conclusion}

This chapter presented the service-oriented, seven-layer system architecture of the TIBSA platform, detailing Edge caching, FastAPI middleware pipelines, asynchronous intelligence layers, PostgreSQL persistence, and threat engines. The Use Case, Sequence, ERD, and Class diagrams confirm the design's structural validation, ensuring that all security assessment, malware analysis, and event correlation workflows operate reliably in a unified full-stack application.
